# Contacts are account-scoped; Subscriptions link Contacts to Publications

A **Contact** is account-scoped, identified by email, and stores cleanable/enrichable attributes (name, company, enrichment metadata, suppression status) — one row per real person across the entire account. A **Subscription** is the explicit link between a Contact and a **Publication**, carrying its own status (`active`, `unsubscribed`, `bounced`, `complained`, `pending_double_optin`). An **Issue** sends to all `active` Subscriptions of its parent Publication.

We rejected per-Publication contact lists because the same person ending up as two rows across Publications is a compliance footgun: an unsubscribe, hard bounce, complaint, or GDPR delete on one row must propagate to every other instance of that person in the account, and a duplicated-rows schema makes that propagation a multi-write transaction that *will* drift out of sync. With a single Contact row and Subscription join, suppression becomes a single write that automatically blocks future sends across all the Contact's Subscriptions — the right thing happens by default. We also rejected a tagged-pool model (one Contact table, Publication membership as tags) because per-Subscription state (double opt-in pending, bounced from one Publication only, paused) doesn't fit cleanly on tags and turns into a parallel state system anyway.

The marginal engineering cost over the simpler shapes is a join table with a status enum. The payoff is that the legally-required operations (global suppression, audit trail, per-Subscription lifecycle) are the default path through the schema rather than features bolted on. Send queries become a straightforward `SELECT contact_id FROM subscriptions WHERE publication_id = ? AND status = 'active'` — no tag parsing, no cross-table dedup.
