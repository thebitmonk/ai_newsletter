# Apollo integration is Publication-scoped; auto-subscribe is opt-in per Publication

Apollo (and any future prospect-sourcing integration) is configured per **Publication**, not per account. Each Publication carries its own search query, its own daily-add cap, and its own boolean `auto_subscribe` flag. A daily job runs per Publication; returned Contacts are upserted into the account-level Contact pool (matched by email, no duplicates created); if `auto_subscribe` is on, an `active` **Subscription** is created tying that Contact to the Publication that found it. If off, the Contact lands in the pool and the owner subscribes manually.

We rejected an account-level Apollo configuration because it forces a "which Publication gets these Contacts?" rule that doesn't exist for any reason other than schema convenience — different Publications want different audiences (an AI engineering Publication and a crypto trading Publication run wildly different prospect searches), and a single account-wide search collapses that. The Publication-scoped model is what the multi-Publication product positioning demands.

We chose `auto_subscribe` as opt-in rather than default-on because the failure mode of getting it wrong once is severe: an owner enables Apollo, doesn't realise auto-sub is on, and the next Issue dispatches to a few hundred cold-pulled prospects who never asked for it. That single incident tanks list reputation, triggers spam-folder placement across the Publication's existing subscribers, and may cost the account its SMTP relationship. A boolean opt-in is a cheap seatbelt that prevents this; the "Apollo as sourcing tool, manual review before subscribe" workflow also drops out naturally.

The daily-add cap denominator is **% of current active Subscriptions on this Publication, clamped to a per-Publication floor and ceiling.** Formula: `add_today = clamp(round(active_subs * pct), floor, ceiling)`. The floor handles new Publications (5% of zero subscribers would otherwise mean Apollo never adds anyone for a brand-new Publication); the ceiling prevents large Publications from burning a year of Apollo credits in a week (5% of 100,000 subscribers is 5,000 contacts a day). Defaults: pct=5%, floor=10, ceiling=100.
