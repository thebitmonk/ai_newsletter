# Bring-your-own sending provider, credentials per-Publication, HTTP API only

The platform does not run a managed email-sending service. Each **Publication** carries its own `sending_config`: provider (`sendgrid` | `mailgun` | `postmark` | `ses`), encrypted API key, `from_address`, `from_name`, `reply_to_address`, `sending_domain`, and `domain_verified` boolean. Sending always goes through the provider's HTTP API via a `SendingProvider` adapter interface — never raw SMTP. Each dispatched recipient is recorded as a **Dispatch** row (Issue × Contact × Subscription) which is the addressable unit for webhook callbacks (bounces, complaints, opens, clicks).

We chose bring-your-own over a managed offering because deliverability is a full-time discipline — IP warmup, ISP relationships, abuse handling, reputation monitoring, jurisdiction-specific compliance — that would consume the company before the product is validated. Becoming "the sending platform" puts us on the hook for every customer's bounce, complaint, and CAN-SPAM violation; deferring to established providers transfers that operational and legal load to organisations who already have those teams. A managed option can be added in v2 once we have product-market fit and know which customer segment is willing to pay the markup.

We chose per-Publication credentials over account-shared because different Publications generally send from different domains, and domain authentication (DKIM/SPF/DMARC) is fundamentally per-domain. Forcing all Publications in an account to share one sending identity would either prevent the multi-Publication use case or leak Publication X's reputation problems into Publication Y's deliverability. Per-Publication isolation matches how real sending works.

We chose HTTP API over raw SMTP because the providers' APIs expose bounce and complaint webhooks (without which the Suppression feedback loop in [ADR-0011](./0011-contact-cleaning-scope-and-defaults.md) cannot operate without postmaster-mailbox polling, which is unworkable), give us idempotency keys, handle rate limiting transparently, support batch sends, and remove the need to operate our own MTA stack.

A hard guard in the **Issue** state machine ([ADR-0007](./0007-issue-state-machine.md)) refuses the `sending` transition if the Publication's `domain_verified` is false — the platform provides an onboarding flow to display required DNS records, poll for their presence, and flip the verification flag. Owners cannot send through an unverified domain.
