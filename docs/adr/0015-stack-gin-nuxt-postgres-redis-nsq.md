# Stack: Gin, Nuxt, PostgreSQL, Redis, NSQ

The platform is built on a fixed four-piece stack: **Gin** for the Go HTTP backend, **Nuxt** (Vue 3) for the frontend, **PostgreSQL** as the primary data store, **Redis** for caching and short-lived coordination state, and **NSQ** for durable background-job queueing.

We chose Gin over net/http stdlib or alternatives (Echo, Chi, Fiber) for an HTTP framework that owners-of-this-repo can pattern-match against existing codebases and tutorial material — it's the most widely-used Go web framework, so contractor onboarding, AI agent generation, and third-party middleware integration all have the lowest friction. The performance and ergonomic differences against Echo/Chi are real but small at this product's scale.

We chose Nuxt over Next.js or a SPA framework because the editor surface and calendar UX benefit from server-rendered shells, Nuxt's auto-imports and file-based routing remove the typical SPA boilerplate, and Vue's reactivity model fits Tiptap's editor state more naturally than React's. Tiptap itself is framework-agnostic but has first-class Vue bindings.

We chose PostgreSQL because every load-bearing decision recorded so far depends on capabilities Postgres has natively and other engines do not match cleanly: `jsonb` with indexing for [ADR-0008](./0008-hybrid-issue-persistence.md)'s Issue body doc, `timestamptz` for [ADR-0014](./0014-per-publication-timezone-utc-storage.md)'s timezone discipline, transactional consistency for [ADR-0009](./0009-account-contacts-with-publication-subscriptions.md)'s Suppression-propagation guarantees, and partial indexes for the recurring "active Subscriptions on Publication X" send query.

We chose Redis for caching and ephemeral coordination (rate-limit counters, idempotency keys for webhook dedup, short-lived distributed locks for "is this Issue already being curated?") — *not* as a queue. Conflating cache and queue concerns into Redis pushes both responsibilities through one process with mismatched durability needs and operational profiles.

We chose NSQ over Redis-based queues (Asynq, river-Postgres), cloud-native services (SQS, PubSub), or workflow engines (Temporal, Inngest) because: (a) NSQ is operationally simple — no central broker, peer discovery via nsqlookupd, durable on-disk message logs; (b) the pipeline's natural shape (Source-poll, Curation, Send) is topic-and-consumer-per-class, which is NSQ's native idiom; (c) it keeps queueing infrastructure under our roof rather than coupled to a cloud provider, preserving the option to self-host or move clouds without a queue migration. Temporal-style workflow engines would be overkill at v1; we can layer durable execution on top of NSQ later if the orchestration grows complex.

The non-negotiable rules these choices imply: (1) timestamps in Postgres are `timestamptz`, never `timestamp`; (2) Redis holds nothing that the system cannot rebuild from Postgres — Redis is a cache, never a source of truth; (3) NSQ messages carry IDs only, never full payloads — consumers re-fetch from Postgres so re-delivery is idempotent against state changes; (4) HTTP routes are versioned (`/api/v1/...`) from day one to avoid breaking-change PR archaeology.
