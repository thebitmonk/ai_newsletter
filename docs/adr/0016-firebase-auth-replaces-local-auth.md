# Firebase Auth replaces local password/session auth

The backend no longer stores passwords or sessions. The User identity column on the **users** table is `firebase_uid` (a `citext unique` column); the legacy `password_hash` column and the entire `sessions` table are dropped. Every authed HTTP request carries a Firebase ID token in the `Authorization: Bearer` header; the Bearer middleware validates it through the Firebase Admin SDK (`firebase.google.com/go/v4`) and, on the first request from a never-seen `firebase_uid`, atomically creates User + Account + `account_members` rows in one transaction.

We chose Firebase for auth because deliverability is not our moat and neither is credential handling — owning email/password storage means owning bcrypt cost tuning, password reset flows, verification email templates, OAuth integrations (Google, Apple, …), MFA enrolment, breach-credential checking, regional data-residency for credentials, and the support load of "I can't sign in." Firebase has solved all of those once. We get the entire surface (email/password, Google, email verification, password reset, future MFA, future social providers) for a single SDK integration and roughly $0 at our scale.

This **modifies** [ADR-0013](./0013-account-user-separated-schema-one-to-one-enforced.md) but does not supersede it. The Account/User separation stays — Account remains the resource-owning workspace, User remains the human identity, `account_members` is still the join with the `role` column reserved for future Editor/Viewer values. Only the User identity column changes: from `(email citext unique, password_hash text not null)` to `(firebase_uid citext unique, email citext nullable, email_verified boolean)`. `email` becomes nullable because Firebase guarantees a UID on every user but not always an email (UID-only providers exist), and loses its unique constraint because Firebase may legitimately have multiple identities sharing an email address (e.g. password + Google login linked to the same person; Firebase handles the linking).

We chose against running our own credential storage with a hosted auth library (e.g. supertokens, ory hydra self-hosted, gothic) because the operational surface — running another stateful service, managing its DB, taking responsibility for any vulnerability disclosure — is the part we're trying to avoid. Firebase's tradeoff is platform lock-in to Google for one specific concern (auth); the rest of the stack is unaffected and the lock-in is bounded because the data we store about users (firebase_uid only) is migratable to any other provider with a similar opaque-UID model.

For local development and CI the Firebase Auth emulator (`firebase-tools`) substitutes for real Firebase: the backend reads `FIREBASE_AUTH_EMULATOR_HOST` and routes Admin SDK calls there instead of `securetoken.google.com`. The Nuxt frontend reads `NUXT_PUBLIC_FIREBASE_AUTH_EMULATOR` and points the JS SDK at the same emulator. Production reads `FIREBASE_PROJECT_ID` + `GOOGLE_APPLICATION_CREDENTIALS` (path to a service-account JSON). A `firebase-emulator` service is added to `docker-compose.yml` so `make up` brings everything required for local auth in one step.

The Bearer middleware's public API (`AccountID(c)`, `UserID(c)`, `CurrentClaims(c)`, `RequireAccountScope`) is unchanged from before this swap; downstream handlers do not need to be touched.
